"The chassis is over-engineering" — a counterpoint

A stress-test of the Cognitive Foundry argument. Where the chassis is right; where it's load-bearing scaffolding that can be removed without consequence.

The thesis I'm pushing back on

From Whitepaper #008

"The chassis consists of four orthogonal artifacts: Contract, Validator, Sensor, Corpus. Each compounds across iteration cycles. Without all four, agent-assisted work degrades to prompt rewrites that lose memory of prior wins."

This counterpoint says: the chassis is over-engineered for most production deployments. Three of the four substrates are optional for the work I've actually seen ship. The contract is load-bearing; the rest is scaffolding that can be removed without the work falling apart.

The case

I've watched four production deployments hit reliable output without all four substrates. Two ran on contract + corpus alone (no formal validator, no runtime sensor). One ran on contract + validator (no corpus, no sensor — small team that read all generations manually). One ran on contract alone, with humans serving as validator and sensor at review time.

The pattern: contracts are necessary; the others are situational. Where the contract is rich, the corpus emerges as a byproduct of code review. The validator is implicit in the type system. The sensor is unnecessary because the team is small enough to read every PR.

The cheap test

If your team has fewer than 8 people shipping to production, your code review IS the sensor. Building a sensor primitive adds tooling overhead for a problem the review cadence already solves. The chassis recommends building it; the team's actual operating conditions don't require it.

What I'd concede

At scale — call it 30+ people committing to the same repo — the implicit substrates stop scaling. Code review becomes lossy. The corpus doesn't emerge cleanly because too many people commit too many variations. Sensors become necessary because no one human can read all the outputs.

The chassis as specified is right for that scale. It's over-engineered for the small-team case where most production agent work actually happens. The whitepaper should make this dependency explicit; as written, it implies the chassis is universally load-bearing.

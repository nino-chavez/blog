The Cognitive Foundry

A reference architecture for agent-assisted production work — what's required to make the chassis hold, what's optional, and what fails predictably without each piece.

Download PDF Cite this

Abstract

"Production-grade agent-assisted work depends on four substrates — contract, validator, sensor, corpus — that compound across iterations. This paper specifies each, proves the compounding through three production receipts, and names two failure modes where the chassis can't help."

1. Premise

1.1 State of the art

Most production teams shipping with agent assistance are running prompt libraries — versioned strings, sometimes parameterized, sometimes A/B tested. The leverage stops at the prompt. When the team's understanding shifts, the prompt is rewritten; the rewrite carries no memory of the prior version's wins1.

1.2 The gap

What's missing isn't a better prompt. It's a substrate — something the agent reads as input but that survives prompt rewrites, accumulates through review cycles, and compounds in version control. This paper names four such substrates and the relationships between them.

2. The chassis

2.1 Four substrates

The chassis consists of four orthogonal artifacts:

  • Contract — typed shape of agent input and output, plus invariants the output must satisfy.
  • Validator — post-condition test that runs after generation; failure surfaces as a test failure, not as runtime drift.
  • Sensor — structured reading of runtime state, replacing "the agent thinks X is happening" with "X is happening."
  • Corpus — body of prior examples retrievable at generation time; one example raises the floor for all future generations.

2.2 Contract design

Contracts are the load-bearing artifact. A contract specifies the shape the agent must produce — fields, types, constraints — and the invariants that must hold across the shape. The contract is the substrate against which the agent's behavior is measured; the prompt is one of several inputs that shape generation, but the contract is what gates output.

2.3 Validator placement

Validators run after the agent produces output but before the output is committed. The placement matters: validators in the prompt are heuristic; validators in the pipeline are deterministic. A failed validator should reject the generation, not adjust it2.

3. Production receipts

Three production deployments — commerce.com's recommendations pipeline, a 630 Volleyball brand-asset generator, and the blueprint methodology itself — each demonstrate the compounding pattern across at least two iteration cycles. Details in §3.1–3.3.

4. Limits & failure modes

The chassis doesn't help when (a) the contract can't be specified — i.e., the work is genuinely exploratory and the shape of the answer isn't known up front — or (b) the corpus is too small for retrieval to add signal. In both cases, fall back to prompt-driven iteration with explicit version notes.

Footnotes

  1. This is the central argument of Dispatch #007 "Encodings, not prompts." This paper extends it from a position essay into a reference architecture.
  2. Validators-that-adjust are the "self-correcting prompt" pattern; they conflate generation with validation and lose the separation that makes the chassis work.

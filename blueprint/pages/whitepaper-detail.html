The Cognitive Foundry

A reference architecture for agent-assisted production work — what's required to make the chassis hold, what's optional, and what fails predictably without each piece.

↳ Companion to: The Cognitive Foundry (post) — the reflection. This is the technical companion.
Executive summary

Agent-assisted production work fails predictably in five places: the contract drifts, the validator skips, the reviewer is the same agent, the runtime can't see what the agent saw, and the failure leaves no encoded capability behind. The chassis below addresses each.

The five-piece chassis — contracts, validators, reviewers, sensors, and encodings — is not a methodology. It's the minimum infrastructure for a non-toy agent loop. Each piece exists because removing it produced a specific failure documented in the appendix.

Three things are intentionally out of scope: prompt engineering (the failure mode is structural, not prompt-shaped), model selection (chassis is model-agnostic), and orchestration frameworks (the chassis is the orchestration). Reach for those after the chassis holds.

The contract — what the agent agreed to do

A contract is a written claim about what the agent will produce and what would falsify the claim. Without one, the agent ships whatever it converged on and the human evaluates after the fact. The evaluation is unanchored — there's no prior commitment to check against.

Contracts in the production chassis carry three fields: the deliverable shape (file path, schema, length budget), the acceptance condition (the test that returns pass/fail), and the falsifier (the specific output that would mean the agent did the wrong thing). The falsifier is the one most often missing in practice; without it, "looks good to me" passes for verification.

Contract shapes by task class

Task classDeliverable shapeAcceptanceFalsifier
Code changeDiff against named filesTests pass; lint cleanTests pass but the change addresses a different bug than stated
Research synthesisMarkdown doc with cited evidenceEach claim has a source linkSources exist but don't support the claim made
Migration scriptIdempotent shell or SQLRe-run produces no changeFirst run succeeds; second run errors or duplicates
API endpointRoute handler + OpenAPI entrySchema matches contractEndpoint exists but returns shape that doesn't match its own docs

Where contracts are missing, the failure is usually scope drift — the agent solved an adjacent problem and the human only noticed in review. The fix isn't "better prompts"; it's a written falsifier that the agent reads before starting.

Validators — proof the contract held

A validator runs the contract's acceptance condition. The validator is independent code — it doesn't ask the agent whether the work is done; it checks the artifact against the contract. The chassis treats agent self-attestation ("done!") as a hypothesis, not a result.

Validator implementations by deliverable type

  • Code — test suite, type-check, lint, dead-code scan
  • Docs — link-checker, broken-citation finder, frontmatter schema validation
  • Data — schema validation, row-count sanity, idempotency probe
  • UI — visual regression, accessibility audit, contract-shape match

The validator runs in the chassis loop, not as a post-hoc review step. When it fails, the agent reads the failure and continues; the human is not in the loop for the validator's verdict. This is the leverage — validators amortize the human's verification work into encoded checks.

Reviewers — a different agent's eyes

A reviewer is a second agent whose only job is to find what the first agent missed. The reviewer doesn't share context with the author agent — it reads the artifact cold and reports what it sees. Same-agent self-review fails predictably: the agent that wrote the artifact has already convinced itself the artifact is correct.

What the reviewer checks that the validator can't

CheckWhy a validator can't do it
The contract was the right contractValidators check the contract; reviewers check whether the contract addressed the actual underlying problem
The diff makes sense as code (not just compiles)Lint sees syntax; reviewer sees idiom and intent
The fix won't break tomorrow's adjacent changeTests cover today's invariants; reviewer reasons about coupling
The artifact is honest about what it doesn't knowValidators check what's there; reviewers check what's missing

Sensors — what the agent could see

A sensor is the agent's grounding in the running system. Without sensors, the agent reasons about a model of the system it built from documentation — which drifts. Sensors close the loop: the agent inspects, doesn't infer.

Common sensor categories: browser drivers (DOM access, console capture), filesystem probes (recursive file listings, content reads), database introspection (live schema, sample rows), and network capture (HTTP responses, header inspection). Each one answers a question the agent would otherwise guess at.

The chassis principle: any time the agent says "probably" about runtime state, a sensor was missing. The fix is to add the sensor, not to ask the agent to be more careful.

Encodings — what the failure left behind

The chassis's compounding leverage comes from encoding each failure as durable infrastructure — a lint rule, a reviewer agent, a sensor primitive, a contract field. The encoding survives the session that produced it. The next initiative inherits the capability for free.

The alternative — patching prompts session-by-session — produces zero compounding. Three years of session-by-session prompt fixes is the same capability surface as one week. Encoding each failure as repo-level infrastructure compounds linearly with use.

The chassis is, in the end, an argument about where to put the work: into prompts that decay or into encodings that accrete. The architecture follows from the argument.

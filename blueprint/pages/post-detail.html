Encodings, not prompts

Where compounding leverage actually lives. Prompt fixes don't accrete — they get rewritten the next time someone touches the file. Encoded environment does.

A prompt is a string. Each time you change it you replace the previous version; nothing about the change carries forward except in someone's memory. An encoding is a structure the agent reads as input — a file, a schema, a contract, a sensor's output. When you change an encoding, the change persists in version control, gets reviewed, and accumulates.

That distinction looks small until you watch what happens after six months of iteration. Teams that tune prompts have the same problems again. Teams that tune encodings have new problems — different ones, the ones immediately upstream of where the first round of work landed.

Why prompts erode

The prompt that worked yesterday gets rewritten today because the failure mode is novel, and the editor's mental model has shifted. The rewrite isn't tracked against the prior version's wins; it's tracked against the new failure. Wins from prior iteration silently degrade. Three months in, the prompt is doing four jobs and none of them well.

Encodings have the opposite property. A schema doesn't get rewritten to address a new failure — it gets extended. A contract isn't replaced — it grows. The work compounds because the substrate compounds.

Counterpoint: prompts ARE encodings (in disguise)

"This essay reifies a distinction the literature already collapsed. A sufficiently rigorous prompt template — versioned, parameterized, schema-validated — is an encoding. The argument is really about discipline, not substrate."

What encodings are, concretely

Four substrates compound: the contract (typed input/output shape the agent must produce), the validator (post-condition test that runs after generation), the sensor (something that observes runtime state and returns a structured reading), and the corpus (the body of prior examples the agent can retrieve from).

None of these are prompts. All of them get reviewed in PRs. All of them survive contact with the next iteration. The work you put into them last quarter is still working this quarter.

Substrate What it encodes How it compounds
Contract Input shape, output shape, invariants Survives prompt rewrites; agent learns the shape, not the wording
Validator Post-condition the agent must satisfy Failure surfaces as test failure, not "the agent did a weird thing"
Sensor Structured reading of runtime state Replaces "the agent thinks X is happening" with "X is happening"
Corpus Examples the agent can retrieve from Adding one example raises the floor for all future generations

What this looks like in production

Watch teams that ship reliably with agent assistance. They are not running prompt libraries. They are running contracts. The prompt for code generation might be three sentences and unchanged for months; the schema the agent generates against is in active development.

The asymmetry is the point. Prompts are the place where the agent's behavior is constrained at the moment of generation. Encodings are the place where the agent's behavior is constrained for everyone, forever — until the encoding is rewritten, which is a deliberate act with a PR attached.

The cheap test

If your prompt changed in the last week and your schema didn't, the leverage is in the wrong place. The fix isn't a better prompt. The fix is an encoding that doesn't require the prompt to change.

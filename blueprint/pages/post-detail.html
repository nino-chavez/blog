The contract is the work

Why agent-assisted work without a written falsifier is unanchored. What the contract carries that the prompt can't.

01
Series: From Prompt to Pattern (the first of eight)

The agent finished. It produced a diff, ran the tests, declared "done." The human reviews and approves or rejects. That's the loop most people are running. It works until it doesn't, and when it stops working, the failure is almost always in the same place: the contract was implicit.

What the agent agreed to do existed in the operator's head, in the prompt's wording, in the assumed context the model inherited from the conversation. None of it was written. When the agent produced something almost right, the operator had to re-derive what "right" was supposed to look like, then evaluate the artifact against that mental model, then accept or reject. The verification cost was the same every iteration. Nothing accreted.

What "contract" means here

Not the legal kind. The engineering kind: a written claim about what the agent will produce and what would falsify the claim. Three fields, minimum:

  1. Deliverable shape — file path, schema, length budget. Concrete enough that an external observer can check it.
  2. Acceptance condition — the test that returns pass/fail. Could be a unit test, a schema validator, a link checker, a manual eyeball check with a specific question.
  3. Falsifier — the specific output that would mean the agent did the wrong thing. This is the field most often missing in practice.

The falsifier is the hard one. "What would mean this is wrong" forces the operator to think about failure modes before the agent runs. Without it, "looks good to me" passes for verification, and the loop has no anchor.

Signal
The contract makes the work checkable by someone other than the operator

If only the person who wrote the prompt can evaluate the result, the work doesn't scale past that person's attention. The contract converts "trust the operator's eye" into "check against the written claim." Different economics.

Why prompts can't do this

Prompts are instructions; contracts are commitments. A prompt says "do X." A contract says "you will produce Y; here is how I'll know if Y is wrong." The first is consumed by the agent. The second persists past the agent's run as the evaluation artifact.

Prompts also decay. The same prompt to the same model on the same task can produce different outputs across runs. Contracts are stable — the falsifier doesn't move. Run the same agent against the same contract a hundred times; you get a hundred artifacts you can each check against the same criterion.

A prompt is consumed by the agent. A contract is what survives.

The minimum-viable contract

For a code change: "Modify file.ts to add a handleX function. The change is correct if the existing test suite passes AND the new test in file.test.ts passes. The change is wrong if it modifies any other file, or if it adds new dependencies."

That's it. Three sentences. The agent reads it before starting and reads it again at the end. The operator reads the diff and the test output, and the question "did the agent do what was agreed" has a deterministic answer.

Most prompts don't carry this. Adding it is the cheapest leverage available in the loop.

What comes next

The contract is the first piece of the five-piece chassis the next post in this series covers — Validators amortize verification. The validator is the encoded form of the contract's acceptance condition. Without the contract, there's nothing to validate. With it, the validator becomes the leverage point that lets the operator stop being in the loop for every iteration.

Most write-ups for posting to Instagram from code assume one account and a token that expires every 60 days. The moment you need to publish to *several* accounts on a schedule, that path falls apart — and the docs don't tell you why, because the answer lives on a different page than the one you started on.

This is the setup that actually works for the multi-account case: one Meta app, one non-expiring token, no App Review, no SaaS scheduler in the middle. The reasoning behind building it instead of renting it is a [separate post](/blog/where-the-api-stops). This one is the runbook.

I'll flag the three things that cost me the most time: the auth-path trap, the manual steps you genuinely cannot automate, and the publish call that returns success while quietly failing.

---

## The Trap: Two Auth Models That Look Like One

Meta exposes two completely different ways to authenticate, and they share enough vocabulary that you can build the wrong one without noticing.

- **Instagram Login** — host `graph.instagram.com`, per-account access tokens you refresh every ~60 days. Built for a single account logging into your app.
- **Facebook Login / Business** — host `graph.facebook.com`, a **System User token** issued from a Business Manager that owns the accounts. One token reaches every account in the portfolio, and it doesn't expire.

If you're publishing to multiple owned accounts on a schedule, you want the second one. I started on the first, got a working single-account token, then had to tear it down when I added accounts. The scopes, the hosts, and the token-exchange endpoints are all different. Pick the Business path first and skip the rework.

<Callout type="signal" title="The decision in one line">
Multiple owned accounts + unattended schedule → one Business Manager, one app, one System User token on `graph.facebook.com`. Single account, user-driven → Instagram Login is simpler. Don't mix them.
</Callout>

---

## Why This Path Has No App Review

The permission you need to publish — `instagram_content_publish` — normally requires **Advanced Access**, which means Meta's App Review queue (think weeks).

The exception: for accounts your Business Manager **owns or is role-connected to**, **Standard Access** is granted automatically. No review, no waiting. That single fact is what makes the owned-accounts case viable on day one. If you were publishing to accounts you *don't* own, you'd be in the review queue and this would be a very different article.

## Hard Limits to Design Around

Learn these before you write code — each one is a constraint that shapes the design, not a runtime surprise.

| Limit | Consequence |
|---|---|
| Business accounts only | Creator-type accounts are rejected by the publish API. Switch each to Business first. |
| 100 published posts / 24h / account | Your scheduler's ceiling. Drip, don't dump. |
| Reels: 9:16, up to ~90s | Validate before upload, not after a failed container. |
| Collaborators must be public | Co-author invites fail for private accounts. |
| Stories have no collaborators | And no publish API for stickers/polls/links at all — that's browser-automation territory. |

## Phase 0 — Account Foundation (Manual, On Purpose)

These steps claim assets and prove ownership, so Meta gates them behind your own login. You cannot script them, and trying wastes more time than doing them by hand. In **business.facebook.com → your Business → Business settings → Accounts → Instagram accounts**:

1. **Add each Instagram account.** Log into each one when prompted — this is the ownership check. If an account only shows up under a Facebook Page, use the Page's **Connect Instagram** instead.
2. **Confirm each is a Business account.** In the Instagram app: Settings → Account type. Creator will not publish via the API.
3. **Link each Instagram account to its Facebook Page.** You need this to resolve the numeric `ig_user_id` later.

---

## Phase 1 — App and System User Token

1. **Create the app.** developers.facebook.com → Create App → **Instagram** product, attached to your business portfolio. (If you already built an Instagram-Login app, abandon it — wrong model.)
2. **Create a System User.** Business settings → Users → System users → Add. Give it the **Admin** role.
3. **Assign assets to the System User:** the app, all the Instagram accounts, and their Pages, each at **Full control**.
4. **Generate the token** for that System User with these scopes:

   ```
   instagram_basic
   instagram_content_publish
   pages_show_list
   pages_read_engagement
   business_management
   ```

   Choose **no expiration**. Copy the token once — you won't see it again.

5. **Store it in a secret manager**, never in the repo. A Worker secret, 1Password, Doppler, Vault — any of them. Everything below reads it from an environment variable:

   ```bash
   export IG_ACCESS_TOKEN="<your-system-user-token>"
   ```

## Resolve the Account IDs

The publish endpoints address each account by a numeric `ig_user_id`, not its handle. One call lists every Page you manage and its linked Instagram business account:

```bash
curl -s "https://graph.facebook.com/v25.0/me/accounts\
?fields=name,id,instagram_business_account{id,username}\
&access_token=$IG_ACCESS_TOKEN" | jq
```

Keep a small registry mapping your own slugs to those IDs:

```json
{
  "brand":     { "handle": "your.brand",  "ig_user_id": "<ig-user-id>" },
  "studio":    { "handle": "your.studio", "ig_user_id": "<ig-user-id>" }
}
```

## The Publish Flow

Publishing a reel is three calls: build a container, wait for the video to transcode, publish the container. The container ID is the thing that matters — hold onto it.

```js
const GRAPH = 'https://graph.facebook.com/v25.0'

async function api(token, path, params, method = 'POST') {
  const body = new URLSearchParams({ ...params, access_token: token })
  const url = `${GRAPH}/${path}`
  const res = method === 'GET'
    ? await fetch(`${url}?${body}`)
    : await fetch(url, { method, body })
  const json = await res.json()
  if (!res.ok || json.error) throw new Error(json.error?.message || JSON.stringify(json))
  return json
}

// 1. Build the container. Returns a creation_id.
async function buildReel(token, igUserId, { videoUrl, caption }) {
  const { id } = await api(token, `${igUserId}/media`, {
    media_type: 'REELS',
    video_url: videoUrl,
    caption,
    share_to_feed: 'true',
  })
  return id
}

// 2. Poll until the video finishes transcoding (images skip this).
async function pollStatus(token, containerId, maxMs = 75000) {
  const deadline = Date.now() + maxMs
  while (Date.now() < deadline) {
    const { status_code } = await api(token, containerId, { fields: 'status_code' }, 'GET')
    if (status_code === 'FINISHED') return 'FINISHED'
    if (status_code === 'ERROR' || status_code === 'EXPIRED') throw new Error(`container ${status_code}`)
    await new Promise((r) => setTimeout(r, 5000))
  }
  return 'IN_PROGRESS' // still transcoding — resume later with the same container
}
```

The `video_url` has to be a **publicly reachable URL** — Meta's servers fetch it. Local file paths don't work. Upload to any public bucket (R2, S3, a CDN) first and pass that URL.

To tag other accounts or send a co-author (Collab) invite, add them to the container call:

```js
const params = {
  user_tags: JSON.stringify([{ username: 'your.studio' }]),
  collaborators: JSON.stringify(['your.studio']),
}
```

Tagging is documented; `collaborators` is community-confirmed but not in Meta's main publishing doc. Treat a rejection as a real, surfaced error — don't drop the invite silently.

## The Gotcha That Double-Posts

This is the one that cost real damage in production, and it's the most transferable lesson here.

The publish call — `POST /{ig}/media_publish` — sometimes returns *"An unexpected error occurred"* **even though the post went live**. If you treat that error as "failed" and re-run, your naive retry builds a *new* container and publishes the *same* video a second time. I shipped exactly that and watched one reel post three times.

Two rules fix it:

**Persist the container ID before you publish, and retry the same one.** Re-publishing an identical `creation_id` is idempotent — Meta won't create a duplicate. So a retry either returns the real media ID (it had actually published) or completes the publish that genuinely hadn't happened. It never doubles.

```js
// Re-publishing the SAME creation_id never duplicates.
async function publishWithRetry(token, igUserId, creationId, tries = 4) {
  let lastErr
  for (let i = 0; i < tries; i++) {
    try {
      const { id } = await api(token, `${igUserId}/media_publish`, { creation_id: creationId })
      return id
    } catch (e) {
      lastErr = e
      // "already published" → fetch and return the existing id instead of retrying
      if (/already.*publish|has already been/i.test(String(e?.message || e))) {
        try { const r = await api(token, creationId, { fields: 'id' }, 'GET'); return r.id }
        catch { return null }
      }
      await new Promise((r) => setTimeout(r, 8000))
    }
  }
  throw lastErr
}
```

**Treat errors as terminal, not as "try again next run."** An item that errored stays errored until you look at it. Only genuinely *pending* items are eligible to post. The failure mode that bit me was a loop that re-queued anything not marked done — so a post that succeeded-but-reported-failure got picked up and posted again on the next tick.

<Callout type="insight">
The reusable rule, far beyond Instagram: when an external API can report failure on a call that actually succeeded, trust the **resource state**, not the **HTTP response**. Make the write idempotent and re-check what's real before you act on the error.
</Callout>

## Persist State Before the Slow Step

Building a container and waiting on transcode can take a minute. If your process dies mid-publish, you don't want to lose the container ID and rebuild from scratch (back to double-posting).

So the order matters: build the container, **save the container ID and mark the item `building`**, *then* poll and publish. On restart, an item still marked `building` resumes from its existing container instead of creating a new one.

```js
const containerId = await buildReel(token, igUserId, item)
item.ig_container_id = containerId
item.status = 'building'
await save(item)            // persist BEFORE the slow poll/publish

if (await pollStatus(token, containerId) !== 'FINISHED') {
  await save(item)          // still transcoding — a later run resumes this container
  return
}
const mediaId = await publishWithRetry(token, igUserId, containerId)
item.status = 'posted'; item.ig_media_id = mediaId
await save(item)
```

## Going Unattended

Running this from a laptop cron means nothing posts when the lid is closed. To make it truly hands-off, move the publish step to a small always-on runner — a Cloudflare Worker on an hourly cron works well: the token becomes a Worker secret, the queue lives in KV (or D1), and the media sits in R2. The publish logic above moves over unchanged.

One thing to guard if you do this: don't let a manual trigger and the scheduled tick run at the same moment against the same queue. Two invocations that both read "pending" before either writes back will each build a container for the same item — the same double-post, by a different cause. A short lock, or simply never triggering manually near the top of the hour, closes it.

## A Note on Versions

The values here — Graph API `v25.0`, the scope names, the 24h limit — are what shipped and worked. Meta rolls API versions on a schedule and deprecates old ones, so check the current version and the [content publishing reference](https://developers.facebook.com/docs/instagram-platform/content-publishing) before you build. The *shape* of the flow — container, poll, publish, idempotent retry — has been stable. The version string and the exact permission names are the parts most likely to have moved by the time you read this.
